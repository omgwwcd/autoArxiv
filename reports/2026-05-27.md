## 1. Scaling, Benchmarking, and Reasoning of Vision-Language Agents for Mobile GUI Navigation

**Topics:** GUI Agent, Vision-Language Models, Mobile GUI Navigation, Reinforcement Learning, Benchmarking
**Authors:** Heng Qu, Yike Liu, Renren Jin, Wenzong Zhang, Pengzhi Gao, Wei Liu, Jian Luan
**Venue / Year:** ICML 2026
**Links:** [Abstract](http://arxiv.org/abs/2605.27134v1) | [PDF](http://arxiv.org/pdf/2605.27134v1.pdf) | [Code](https://github.com/xiaomi-research/guievalkit)

### 1. 一句话结论

强化微调（DAPO）在移动GUI导航中始终优于监督微调，尤其在域外场景中，且性能随训练数据量对数线性增长。

### 2. 为什么重要

该工作提供了大规模中文移动GUI数据集HyperTrack和统一评测工具GUIEvalKit，系统研究了数据规模与强化学习对VLM代理性能的影响，为移动GUI导航代理的开发与评估奠定了坚实基础。

### 3. 研究问题

1. 训练数据规模如何影响监督微调和强化微调下VLM代理在移动GUI导航中的性能？
2. 交互历史和推理能力如何影响任务完成？

### 4. 背景与问题设定

移动GUI导航要求AI代理整合多模态感知、长程推理和序列决策。现有数据集规模小、以英文为主，缺乏对中文应用覆盖和标准化评测工具。数据规模对监督和强化微调的影响尚不明确，推理能力的作用也未系统研究。

### 5. 方法概览

- 任务 / 环境: 移动GUI导航任务，基于674个中文Android应用，包含16080个真实世界任务，动作空间包括OPEN、CLICK、SCROLL、TYPE、STOP。
- 变量 / 干预设计: 使用HyperTrack数据集，创建10个不同规模（16至8192个episode）的训练子集，分别采用监督微调（SFT）和基于DAPO的强化微调（RL）训练UI-TARS-1.5-7B模型。
- 评价指标: 步骤级：类型匹配率（Type Match）和精确匹配率（Exact Match）；回合级：进度（Progress）和成功率（Success Rate）。
- 模型比较: 比较SFT与RL在不同数据规模下的性能，并在域内、未见应用、未见设备、未见应用与设备四种测试集上评估。额外使用Qwen3-VL-8B-Thinking验证跨骨干和奖励函数的可扩展性。
- 我的理解: 该方法通过系统控制数据规模并对比两种微调范式，揭示了RL在数据扩展中的优势，特别是域外泛化能力。奖励设计结合动作类型和参数匹配，并引入高斯空间奖励提供更密集的点击反馈。

### 6. 关键发现

- 性能随训练数据对数线性增长: SFT和RL模型的类型匹配和精确匹配准确率均随训练episode数量的对数近似线性提升。
- 强化微调在域外场景中显著优于监督微调: 在域内设置中RL提升有限，但在未见应用等域外场景中，RL模型性能大幅领先SFT，表明RL具有更强的泛化能力。
- 专用GUI代理优于通用VLM: UI-TARS和AgentCPM-GUI等专用代理在多数基准上表现优于通用VLM，凸显任务特定训练的重要性。
- 思考模式并非总是有益: 对于支持可控思考的模型（如MiMo-VL、GUI-Owl），禁用思考模式反而获得更高的步骤匹配率，表明显式推理痕迹可能引入冗余或干扰精确动作执行。

- 半在线评估虽改进了上下文对齐，但仍非完全在线评估，无法替代真实环境测试。

## 2. IPIBench: Evaluating Interactive Proactive Intelligence of MLLMs under Continuous Streams

**Topics:** GUI Agent, Multimodal Large Language Models, Streaming Video Understanding, Proactive Intelligence
**Authors:** Jinzhao Li, Yinuo Chen, Wenxuan Song, Yijia Lei, Yichi Zhang, Honglei Yan, Panwang Pan, Miao Liu
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2605.27074v1) | [PDF](http://arxiv.org/pdf/2605.27074v1.pdf) | [Code](https://lijinzhao30.github.io/IPIBench/)

### 1. 一句话结论

IPIBench是首个评估多模态大模型在流式视频中交互式主动智能的基准，包含1,831视频和3,738问答对，揭示模型在主动触发稳定性和多轮协调上的严重不足，而训练无关的IPI-Agent框架在主动监控、任务管理和交织任务上分别平均提升约15、12和10个百分点（基于表2中Gemini 3 Pro数据估算）。

### 2. 为什么重要

现有基准多聚焦单轮反应或主动任务，忽略真实场景中用户动态增删改主动请求与反应查询交织的复杂交互。IPIBench填补了这一空白，为构建真正可用的流式AI助手提供了关键评估工具和解决方案。

### 3. 研究问题

1. 多模态大模型在连续流式视频中能否稳定、及时地触发主动响应？
2. 模型能否在主动监控、任务管理与反应查询交织的多轮交互中有效协调行为？

### 4. 背景与问题设定

现有MLLMs在反应式问答上表现优异，但流式AI助手（如可穿戴设备、家庭机器人）需持续感知并主动推理。已有基准仅评估单轮主动或反应任务，忽略了用户可能随时添加、修改、取消主动请求，并穿插反应查询的动态多轮场景。

### 5. 方法概览

- 任务 / 环境: 连续流式视频输入，模型仅能访问当前及历史帧，需实时决策是否触发主动响应、更新任务或回答反应查询。
- 变量 / 干预设计: IPIBench包含三大类任务：主动监控（时机、理解、重复触发）、主动任务管理（取消、修改、多任务）、交织反应-主动请求（反应后主动、反应中主动、反应转主动）。
- 评价指标: 各子任务正确率（%），综合平均分。
- 模型比较: 评估了闭源模型GPT-4o、GPT-5.4、Gemini 2.5 Pro、Gemini 3 Pro，以及开源模型如LLaVA-OneVision等，并对比人类表现（人类平均96.67%）。
- 我的理解: 该基准通过精心设计的视频片段和人工标注的触发时刻，模拟了真实流式交互中用户指令的动态变化，迫使模型同时处理时间精度、任务切换和查询类型识别。

### 6. 关键发现

- 现有MLLMs主动触发不稳定: 在单轮主动监控任务中，模型常过早或延迟响应。例如，Gemini 3 Pro在Proactive Timing子任务上正确率仅43.49%，Proactive Understanding为24.90%，Repeated Proactiveness为18.40%，平均28.93%，远低于人类94.67%。GPT-4o平均30.09%，同样表现不佳。
- 多轮交互协调能力弱: 在Proactive Task Management任务中，Gemini 3 Pro平均正确率仅20.22%（Cancel 19.44%，Modify 21.84%，Multi-task 19.39%）；在Interleaved Reactive-Proactive任务中平均23.04%。模型难以正确执行取消、修改操作，也无法在反应查询后继续监控主动任务。
- IPI-Agent显著提升性能: 提出的训练无关框架通过交互控制策略和时间门控机制，在所有任务上一致提升基础MLLMs表现。例如，在Proactive Monitoring任务上，Gemini 3 Pro从28.93%提升至约44%（提升约15个百分点，基于表2数据估算）；在Proactive Task Management上从20.22%提升至约32%（提升约12个百分点）；在Interleaved Reactive-Proactive任务上从23.04%提升至约33%（提升约10个百分点）。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: 未来流式AI助手需内置统一的交互策略，而非仅优化单轮问答；主动触发稳定性是实际部署的关键瓶颈。
- 对 Skill: 模型需同时掌握时间感知（何时触发）、任务管理（增删改）和查询类型识别（反应vs主动）。
- 对 Memory: 需要显式的交互记忆和主动任务记忆，以支持长期监控和上下文增强。
- 对评测: 基准设计应包含动态指令变化和交织查询，避免单轮静态评估的局限性。

### 11. 局限性

- IPIBench基于公开数据集构建，视频场景多样性有限，可能未覆盖所有真实流式交互模式。
- IPI-Agent为训练无关框架，依赖预训练MLLM和嵌入模型，性能受限于基础模型能力。
- 基准仅评估开放生成任务，未涉及更细粒度的行为分析（如响应延迟分布）。

### 12. 我的看法 / 个人笔记

该工作精准定位了当前MLLMs在流式交互中的关键缺陷，提出的基准和框架具有实用价值。未来可探索端到端训练方法以进一步提升协调能力，并扩展至更复杂的多模态输入（如音频）。

### 14. 总结

IPIBench是首个评估MLLMs在流式视频中交互式主动智能的基准，涵盖主动监控、任务管理和交织反应-主动请求三大类任务，包含1,831视频和3,738问答对。实验揭示模型在主动触发稳定性和多轮协调上的严重不足（如Gemini 3 Pro在主动监控任务上平均仅28.93%），而提出的IPI-Agent框架通过交互控制策略和时间门控机制显著提升性能（主动监控提升约15个百分点，任务管理提升约12个百分点，交织任务提升约10个百分点），为构建实用流式AI助手提供了评估工具和解决方案。

## 3. LocateAnything: Fast and High-Quality Vision-Language Grounding with Parallel Box Decoding

**Topics:** 视觉定位, 目标检测, 多模态大模型, 并行解码
**Authors:** Shihao Wang, Shilong Liu, Yuanguo Kuang, Xinyu Wei, Yangzhou Liu, Zhiqi Li, Yunze Man, Guo Chen, Andrew Tao, Guilin Liu, Jan Kautz, Lei Zhang, Zhiding Yu
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2605.27365v1) | [PDF](http://arxiv.org/pdf/2605.27365v1.pdf) | [Code](https://github.com/...)

### 1. 一句话结论

LocateAnything 通过并行框解码（PBD）将边界框作为原子单元一步解码，在保持几何一致性的同时大幅提升吞吐量和定位精度。

### 2. 为什么重要

现有 VLM 将 2D 框序列化为 1D 令牌逐 token 生成，既破坏几何耦合又造成推理瓶颈。LocateAnything 首次将多 token 预测与结构化单元对齐，实现速度与精度的双赢，对实时交互和具身智能系统意义重大。

### 3. 研究问题

1. 如何将多 token 预测（MTP）与边界框的结构化特性对齐，以同时提升解码速度和定位质量？
2. 在并行解码中，如何设计训练策略和推理机制以平衡吞吐量与鲁棒性？

### 4. 背景与问题设定

视觉语言模型（VLM）常将目标检测和定位建模为坐标 token 生成问题，将 2D 框序列化为多个 1D token 逐 token 解码。这种方式与框几何的耦合结构不匹配，且造成推理瓶颈。现有 MTP 方法虽能并行预测多个 token，但通常结构无关，将输入视为通用 token 流，导致模型学习到跨框边界的不可靠模式，引入虚假关联并降低精度。

### 5. 方法概览

- 任务 / 环境: 统一的视觉定位与检测任务，包括指代表达理解、目标检测、GUI 元素定位等。
- 变量 / 干预设计: 提出并行框解码（PBD），将每个边界框（或点）视为原子单元，在单步中并行预测完整坐标集 (x1,y1,x2,y2)。训练时采用双序列联合优化：标准 NTP 序列保持因果推理能力，块级 MTP 序列实现框对齐预测。通过特殊注意力掩码隔离两序列，同时允许共享上下文。
- 评价指标: 解码吞吐量（throughput）、高 IoU 定位质量（如 mAP@0.5/0.75）、速度-精度前沿。
- 模型比较: 与基于 NTP 的基线（如 Shikra、Qwen-VL）以及现有 MTP 方法对比，在多个基准（布局定位、长尾检测、GUI 定位）上均取得更优的吞吐量和定位精度。
- 我的理解: 核心创新在于将 MTP 的并行预测与框的结构化单元对齐，而非随机 chunk。通过双序列训练和混合推理模式，既保留了因果推理的稳定性，又实现了并行解码的高效性。

### 6. 关键发现

- 并行框解码提升吞吐量和定位精度: PBD 相比 NTP 实现高达 2.5 倍解码吞吐量提升，同时在多个基准上改善高 IoU 定位质量，表明框对齐的并行解码优于逐 token 生成。
- 混合模式平衡速度与鲁棒性: Hybrid Mode 默认使用快速 MTP，当检测到格式不规则或空间歧义时回退到 NTP，在保持大部分速度增益的同时减少最坏情况下的失败。
- 大规模数据引擎提升多样性: 构建的 LocateAnything-Data 包含超过 1.38 亿训练样本，覆盖检测、指代、GUI、布局、OCR 等多种任务，显著增强高精度定位的数据多样性。

- 大规模数据引擎的构建依赖多源数据集，可能存在标注噪声和领域偏差。

## 4. Can Retrieval Heads See Images? Multimodal Retrieval Heads in Long-Context Vision-Language Models

**Topics:** 多模态检索头, 长上下文视觉语言模型, 注意力机制可解释性
**Authors:** Aaron Branson Cigres Li, Zhaowei Wang, Yu Zhao, Yiming Du, Haobo Li, Xiyu Ren, Ginny Wong, Simon See, Lishu Luo, Haodong Duan, Pasquale Minervini, Yangqiu Song
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2605.27243v1) | [PDF](http://arxiv.org/pdf/2605.27243v1.pdf) | [Code](https://github.com/ab-cli/MMRetHead)

### 1. 一句话结论

本文提出基于注意力分数的多模态检索头检测方法，发现LVLM中仅4.4-10.2%的注意力头贡献50%的检索分数，且这些头对长上下文检索和推理具有因果重要性。

### 2. 为什么重要

该工作首次将检索头概念从纯文本扩展到多模态场景，揭示了LVLM中稀疏且因果重要的注意力机制，为理解模型行为、提升长上下文多模态任务性能提供了新视角和诊断工具。

### 3. 研究问题

1. LVLM中是否存在类似LLM的检索头，且能通过注意力分数检测？
2. 多模态检索头是否具有稀疏性、内在性和因果重要性？
3. 检索头在不同模态和上下文长度下如何动态变化？

### 4. 背景与问题设定

大型视觉语言模型（LVLM）需处理长上下文（如文档、视频、智能体轨迹），但现有检索头检测方法基于文本复制（copy-paste），不适用于视觉证据。本文引入多模态检索头（MMRetHead），通过计算问题token到证据token的注意力分数，并采用空问题校准去除偏差，检测文本和视觉证据的检索头。

### 5. 方法概览

- 任务 / 环境: 使用MM-NIAH数据集（来自MMLongBench），包含文本检索、图像检索、渲染文本检索和相同图像检索任务，上下文长度从8K到128K，每个条件20个示例，每个示例6个深度位置。
- 变量 / 干预设计: 通过零掩码移除检测到的检索头（top 5%），评估对MM-NIAH、长文档VQA（MMLongBench-Doc、SlideVQA）和推理基准（MMMU、MathVista等）的影响，并与随机头掩码对比。
- 评价指标: 检索分数质量（正分数质量占比）、准确率、Recall@1。
- 模型比较: 比较Qwen2.5-VL（7B, 32B）、Qwen3-VL（8B, 32B）、Gemma3（12B, 27B）系列模型，以及检索头掩码与随机头掩码的效果。
- 我的理解: 该方法通过计算问题token到证据token的注意力分数，并校准空问题基线，识别出稀疏且因果重要的检索头。关键创新在于将注意力分数直接用于多模态证据，而非依赖文本复制，并通过空问题校准消除位置偏差。

### 6. 关键发现

- 多模态检索头稀疏且因果重要: 仅4.4-10.2%的注意力头贡献50%的检索分数；掩码前5%检索头导致MMLongBench-Doc从48.2%降至5.7%，SlideVQA从71.2%降至8.9%，而随机掩码仅降至32.2%和52.6%。
- 检索头具有内在性和动态适应性: 文本检索头在多模态适配后大部分保留（Qwen3-VL-8B中46/58共享，Gemma3-12B-IT中34/39共享）；图像检索头随上下文长度变化更大，且长上下文时向深层移动（8K时平均层33.1，128K时平均层52.4）。
- 检索头可迁移至文档检索任务: 无需额外训练，使用检索头分数在MMDocIR上提升Recall@1 7.7/7.4（页面检索）和6.3/6.8（布局检索），超过最强基线。
- 跨长度因果迁移: 8K检测的检索头在128K上下文中仍具因果重要性（图像检索准确率降至0.0%，文本检索降至9.1%），反之亦然。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: 检索头可解释智能体在长轨迹中定位关键信息的行为，指导设计更高效的注意力机制或模型压缩。
- 对 Skill: 多模态检索头可能构成模型的核心技能（如文档检索、视觉问答），其稀疏性提示可针对性优化或加速。

该工作系统性地将检索头概念扩展到多模态，方法简洁有效，实验充分。未来可探索检索头在更复杂任务（如视频理解、多轮对话）中的角色，以及如何利用其稀疏性进行模型压缩或加速。

## 5. MUSE-Autoskill: Self-Evolving Agents via Skill Creation, Memory, Management, and Evaluation

**Topics:** LLM Agent, Skill Lifecycle, Self-Evolving Agent, Automatic Skill System, GUI Agent
**Authors:** Huawei Lin, Peng Li, Jie Song, Fuxin Jiang, Tieying Zhang
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2605.27366v1) | [PDF](http://arxiv.org/pdf/2605.27366v1.pdf)

### 1. 一句话结论

MUSE-Autoskill 提出统一的技能生命周期框架（创建、记忆、管理、评估、精炼），使 LLM 代理持续自我进化，在 SkillsBench 上达到 68.4% 准确率（较无技能基线提升 15.21 个百分点），且自生成技能可跨代理迁移。

### 2. 为什么重要

现有技能创建方法（如 Voyager、AutoSkill）仅覆盖部分生命周期，将技能视为孤立静态工件，缺乏复用性和长期改进。MUSE 首次将技能视为长期演化的资产，引入技能级记忆（跨任务积累每个技能的使用经验）和单元测试驱动的自动精炼，实现了技能全生命周期管理。与现有方法相比（见表1），MUSE 是唯一同时覆盖创建、记忆、管理、评估、精炼五个阶段且无需训练的方法，为构建可积累、可测试、可迁移的代理能力基础设施提供了系统方案。

### 3. 研究问题

1. 如何设计统一的技能生命周期框架，使 LLM 代理能够持续创建、复用和精炼技能？
2. 技能级记忆能否提升技能跨任务的复用效果和适应性？
3. 自生成技能能否超越人类编写的技能，并跨代理迁移？

### 4. 背景与问题设定

LLM 代理依赖可复用的技能解决复杂任务。现有方法（如 Voyager、AutoSkill）仅覆盖部分生命周期，存在四个关键缺口：创建与使用脱节（技能在无运行时上下文中生成）、缺乏结构化技能级记忆（无法跨任务积累经验）、技能未经单元测试验证（可靠性低）、长任务中上下文处理不佳（信息丢失或溢出）。本文提出技能应作为长期演化的资产，形式化为创建、记忆、管理、评估、精炼五个阶段。

### 5. 方法概览

- 任务 / 环境: SkillsBench 基准，包含 51 个真实世界任务，涵盖科学工程、数据分析、文档处理、运维规划四个超级领域，在标准化 Docker 环境中由自动验证器评分。
- 变量 / 干预设计: 对比三种 GPT-5.5 驱动的代理（Codex、Hermes、MUSE），每种代理分别在无技能、有人类技能、有自生成技能条件下测试。MUSE 还测试了技能跨代理迁移（将 MUSE 生成的技能注入 Hermes）。
- 评价指标: 任务准确率（Accuracy），即自动验证器判定任务成功的比例。
- 模型比较: MUSE-Autoskill 在有人类技能条件下总体准确率 68.4%，高于 Codex（67.3%）和 Hermes（61.2%），较无技能基线提升 15.21 个百分点。在自生成技能条件下，35 个成功生成技能的任务上准确率达 87.94%，超越人类技能上限。
- 我的理解: MUSE 的核心创新在于：1) 技能创建嵌入代理运行时循环（通过内置 skill_create 工具），消除创建-使用脱节；2) 引入技能级记忆，为每个技能维护一个经验笔记（如超时处理、偏好设置），跨任务积累并影响后续调用；3) 单元测试驱动的评估：创建技能后自动运行 tests/ 目录下的测试，仅通过后注册到技能库，失败则触发 update_skill 精炼；4) 自适应上下文压缩和跨会话状态持久化，支持长任务。这使得技能成为外部化、可测试、可转移的资产。

### 6. 关键发现

- MUSE 在 SkillsBench 上取得最佳准确率: 在三种 GPT-5.5 代理中，MUSE 在有人类技能条件下总体准确率 68.4%，比无技能基线提升 15.21 个百分点，在 4 个超级领域中的 3 个上领先。
- 自生成技能在成功创建的任务上超越人类技能上限: 在 35 个成功生成技能的任务上（技能创建成功率 35/51），MUSE 自生成技能准确率达 87.94%，超过人类技能的最佳表现。
- 技能可跨代理迁移: 将 MUSE 生成的技能注入 Hermes 代理，使其准确率提升 10.51 个百分点，缩小了与人类技能 Hermes 差距的 79%，证明技能是外部化知识资产而非代理特定行为。

### 13. 这篇工作引用的近期高影响力相关文章

- Voyager: An Open-Ended Embodied Agent with Large Language Models
  为什么重要：开创性工作，首次在 Minecraft 中实现自动技能创建和库维护，但仅覆盖部分生命周期。
  核心贡献：提出基于 LLM 的代理通过自我验证和迭代提示不断扩展可执行代码技能库。
  与当前论文关系：MUSE 将其技能生命周期思想扩展到通用代理，并补全了记忆、评估、精炼等缺失阶段。
- AutoSkill: Automatic Skill Creation and Reuse for LLM Agents
  为什么重要：将 Voyager 范式推广到通用代理，实现技能从对话轨迹中派生和复用。
  核心贡献：提出模型无关的技能插件层，从交互轨迹中提取技能。
  与当前论文关系：MUSE 指出 AutoSkill 缺乏结构化技能级记忆和单元测试驱动评估，并在此基础上增加了这些组件。
- SkillOS: Learning to Curate Skill Repositories for LLM Agents
  为什么重要：通过训练可迁移的策展器来管理技能库，展示了技能跨骨干模型的可移植性。
  核心贡献：冻结执行器，训练可迁移的策展器更新外部技能库。
  与当前论文关系：MUSE 强调技能本身作为转移单元，而非策展器，并首次通过跨代理实验验证了技能的直接可迁移性。

### 14. 总结

MUSE-Autoskill 提出统一的技能生命周期框架（创建、记忆、管理、评估、精炼），使 LLM 代理能够持续自我进化。在 SkillsBench 上，MUSE 达到 68.4% 准确率（较无技能基线提升 15.21 个百分点），自生成技能在成功创建的任务上超越人类技能上限（87.94%），且技能可跨代理迁移。该工作强调了将技能视为长期演化资产的重要性，为构建可积累、可测试、可转移的代理能力基础设施提供了系统方案。
