## 1. SpatialClaw: Rethinking Action Interface for Agentic Spatial Reasoning

**Topics:** GUI Agent
**Authors:** Seokju Cho, Ryo Hachiuma, Abhishek Badki, Hang Su, Byung-Kwan Lee, Chan Hee Song, Sifei Liu, Subhashree Radhakrishnan, Seungryong Kim, Yu-Chiang Frank Wang, Min-Hung Chen
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.13673v1) | [PDF](http://arxiv.org/pdf/2606.13673v1.pdf) | [Code](https://github.com/SeokjuCho/SpatialClaw)

### 1. 一句话结论

SpatialClaw 通过将代码作为动作接口，在持久化 Python 内核中实现灵活组合与迭代修正，在 20 个空间推理基准上平均准确率 59.9%，超越此前最佳方法 11.2 个百分点。

### 2. 为什么重要

该工作重新定义了工具增强型空间推理智能体的动作接口设计，证明代码作为动作接口相比单次执行或结构化工具调用能显著提升开放、复杂 3D/4D 空间推理能力，且无需针对模型或基准进行适配。

### 3. 研究问题

1. 动作接口的设计如何影响工具增强型智能体在开放空间推理任务中的能力？
2. 相比单次代码执行和结构化工具调用，代码作为动作接口能否带来更灵活的组合与迭代修正能力？

### 4. 背景与问题设定

视觉语言模型在空间推理（如判断物体位置、运动、3D 关系）上仍存在根本性挑战。工具增强型智能体通过集成专用感知模块来弥补 VLM 的不足，但其有效性受限于调用这些工具的动作接口。现有接口要么是单次代码执行（需在观察中间结果前提交完整策略），要么是结构化工具调用（灵活性有限，难以自由组合操作）。两者均难以支持开放、复杂的 3D/4D 空间推理。

### 5. 方法概览

- 任务 / 环境: 20 个空间推理基准，涵盖单图像、多视图、视频/4D、通用空间推理及通用视频理解等任务。
- 变量 / 干预设计: 提出 SpatialClaw 框架，采用代码作为动作接口：维护一个持久化 Python 内核，预加载输入帧、感知工具和科学计算库；VLM 智能体每步生成一个可执行代码单元，并基于所有先前输出（文本、变量、可视化）进行下一步决策。
- 评价指标: 各基准上的准确率，以及平均准确率。
- 模型比较: 与无工具基线、SpaceTools-Toolshed、pySpatial 等对比；在 Qwen 和 Gemma4 两个模型家族（27B-397B）上验证，无需任何基准或模型特定适配。
- 我的理解: SpatialClaw 的核心创新在于将代码视为一个编排空间，而非一次性程序或工具调度接口。通过持久化内核和迭代循环，智能体可以逐步构建、检查并修正空间分析，例如先分割物体、重建场景，再根据中间可视化结果调整距离计算方法（如从质心距离改为最近点距离）。

### 6. 关键发现

- 代码作为动作接口显著提升空间推理性能: SpatialClaw 在 20 个基准上平均准确率 59.9%，比此前最佳空间智能体（SpaceTools-Toolshed）高出 11.2 个百分点，且在动态 4D 视频推理和多视图推理等需要链式几何计算的任务上提升最大。
- 性能提升跨模型家族和参数规模一致: 在 Qwen 和 Gemma4 两个模型家族（27B-397B）上均观察到一致改进，无需修改系统提示、工具集或基准特定工程，证明动作接口本身驱动了增益。
- 灵活组合与中间验证是关键: 通过对比实验和消融研究，SpatialClaw 的接口通过灵活组合感知原语和中间结果可视化验证，实现了最强的泛化能力；即使移除所有预定义工具包装器，增益仍然存在。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: 代码作为动作接口为智能体系统设计提供了新范式：持久化内核和迭代执行使智能体能够逐步构建复杂分析，适用于需要组合多种工具和数值计算的开放任务。
- 对 Skill: 智能体需要具备代码生成和调试能力，以及将感知工具输出与科学计算库（如 NumPy、SciPy）灵活组合的技能。
- 对 Memory: 持久化内核自动维护中间变量状态，无需显式记忆管理，降低了智能体对长期记忆的依赖。
- 对评测: 评估应覆盖多种空间推理任务（静态/动态、单视图/多视图），并关注接口设计对泛化能力的影响，而非仅关注模型或工具集。

### 11. 局限性

- 依赖 VLM 的代码生成能力，若模型代码生成质量低则性能受限。
- 持久化内核和迭代循环增加了计算开销和延迟。
- 当前工具集主要针对空间推理，扩展到其他领域可能需要重新设计原语。

### 12. 我的看法 / 个人笔记

SpatialClaw 通过将代码作为动作接口，巧妙地将智能体的推理过程从一次性提交转变为迭代构建与验证，显著提升了复杂空间推理的准确性和鲁棒性。其跨模型、跨基准的泛化能力表明，接口设计本身是智能体能力的关键因素，而不仅仅是工具或模型的选择。未来可探索更高效的代码生成策略和更丰富的感知原语。

### 13. 这篇工作引用的近期高影响力相关文章

  核心贡献：提供一组预定义感知工具和结构化调用接口。

### 14. 总结

SpatialClaw 提出代码作为动作接口，通过持久化 Python 内核和迭代执行循环，使 VLM 智能体能够灵活组合感知工具和科学计算库，逐步构建、检查并修正空间分析。在 20 个空间推理基准上平均准确率 59.9%，超越此前最佳方法 11.2 个百分点，且跨模型家族和参数规模一致提升，无需任何适配。该工作强调了动作接口设计在智能体能力中的核心作用。

## 2. IterCAD: An Iterative Multimodal Agent for Visually-Grounded CAD Generation and Editing

**Topics:** GUI Agent, Computer-Aided Design, Multimodal Agent
**Authors:** Tao Hu, Jiaxin Ai, Licheng Wen, Xueheng Li, Shu Zou, Siqi Li, Nianchen Deng, Xinyu Cai, Hongbin Zhou, Pinlong Cai, Daocheng Fu, Yu Yang, Hairong Zhang, Botian Shi, Xuemeng Yang
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.13368v1) | [PDF](http://arxiv.org/pdf/2606.13368v1.pdf)

### 1. 一句话结论

IterCAD 提出闭环多轮交互 CAD 智能体，通过渐进式 SFT 和几何感知强化学习（含 GVPM），在 IterCAD-Bench 上代码可执行率提升至 92.3%（基线 CAD-Coder 为 68.1%），几何精度 AUC-TR 达 0.87（基线为 0.52）。

### 2. 为什么重要

现有 CAD 自动化方法多为开环一次性生成，与工程师迭代式工作流程不匹配。IterCAD 通过闭环迭代和几何感知优化弥合了这一差距，并提出了消除幸存者偏差的 CD-TR 曲线评估标准，为工程级 CAD 自动化提供了新范式。

### 3. 研究问题

1. 如何实现闭环、多轮交互的 CAD 生成与编辑？
2. 如何设计训练策略以提升代码可执行性和几何保真度？
3. 如何建立消除幸存者偏差的评估指标？

### 4. 背景与问题设定

计算机辅助设计（CAD）在现代制造中至关重要，但现有自动化方法（如 CAD-Coder、CADrille）主要依赖开环一次性生成，缺乏迭代自校正能力。多轮框架受限于粗粒度反馈：点云距离仅捕捉全局几何差异，编译器反馈仅验证语法可执行性，均无法定位具体参数错误。现有基准测试（如 Text2CAD）偏向简单模式（如草图-拉伸），且评估协议存在幸存者偏差，仅对成功执行的代码计算几何指标，导致性能失真。

### 5. 方法概览

- 任务 / 环境: 多轮交互环境，智能体与可执行 CAD 沙箱（基于 CadQuery）交互，沙箱返回编译器日志、执行结果和视觉反馈（正交投影及尺寸标注）。任务包括 Drawing-to-Code、Text-to-Code 和 Interactive Editing。
- 变量 / 干预设计: 两阶段训练：1) 渐进式冷启动 SFT，先基于专家轨迹（由 Qwen3-VL-235B 生成）建立基础能力，再通过策略自展开收集失败案例并由教师模型生成修正轨迹进行二次微调；2) 几何感知强化学习（GSPO），结合几何奖励（基于 Chamfer Distance 的 RCD）、格式奖励（Rfmt）和进度奖励（Rprog），并引入几何-可行前缀掩码（GVPM），通过检测连续 K 次运行时错误（fexec）或几何停滞（fstall）确定前缀边界，仅对前缀内 token 计算损失，并对被掩码轨迹施加单侧优势钳制（ˆAg ← max(ˆAg,0)），解决信用分配问题。
- 评价指标: 提出 Chamfer Distance Tolerance-Recall (CD-TR) 曲线及其 AUC-TR 指标，将失败生成（CD→∞）纳入计算，统一衡量代码有效性和几何精度，消除幸存者偏差。
- 模型比较: 与 CAD-Coder、CADrille 等基线对比，IterCAD 在 IterCAD-Draw 任务上代码可执行率达 92.3%（CAD-Coder 为 68.1%，CADrille 为 71.5%），AUC-TR 达 0.87（CAD-Coder 为 0.52，CADrille 为 0.58）。在 IterCAD-Edit 任务上，编辑成功率提升至 85.6%（基线为 45.2%）。
- 我的理解: 该方法将 CAD 建模视为闭环迭代过程，智能体通过多轮交互逐步优化代码，利用多视图工程图纸作为空间锚点，结合编译器、执行和视觉反馈进行修正。GVPM 通过识别不可恢复的错误循环或几何停滞，防止下游失败污染早期正确步骤的梯度，是解决长程信用分配的关键设计。

### 6. 关键发现

- 闭环迭代显著提升 CAD 生成质量: IterCAD 通过多轮“生成-验证-修正”循环，有效修复执行错误和几何缺陷。在 IterCAD-Draw 任务中，最终代码可执行率达 92.3%，几何精度 AUC-TR 达 0.87，均显著优于基线（CAD-Coder 可执行率 68.1%，AUC-TR 0.52）。
- GVPM 解决长程信用分配问题: GVPM 通过检测连续 K 次运行时错误（fexec）或几何停滞（fstall）确定前缀边界，仅对前缀内 token 计算损失，并对被掩码轨迹施加单侧优势钳制（ˆAg ← max(ˆAg,0)），确保早期正确步骤不被下游失败惩罚。消融实验显示，移除 GVPM 后 AUC-TR 下降 0.15。
- CD-TR 曲线消除幸存者偏差: 新指标将失败生成（CD→∞）纳入计算，更真实地反映智能体在工程场景下的鲁棒性。传统指标仅计算成功样本，而 CD-TR 曲线通过 AUC-TR 统一衡量代码有效性和几何精度。在 IterCAD-Bench 上，传统指标高估基线性能约 30%，而 CD-TR 曲线提供了更公平的对比。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: 展示了多模态智能体在复杂工程任务中的闭环交互能力，可推广至其他需要迭代优化的设计领域（如电路设计、建筑设计）。GVPM 的信用分配机制对长程任务具有通用价值。
- 对 Skill: 智能体需具备多模态理解（图纸、文本）、代码生成、错误诊断和几何推理等综合技能。IterCAD 通过两阶段训练有效整合了这些能力。
- 对 Memory: 历史交互记录（包括先前代码和反馈）对证据驱动的修正至关重要，GVPM 通过掩码机制有效管理长程依赖，防止下游失败污染早期梯度。
- 对评测: CD-TR 曲线为评估闭环系统提供了更公平的指标，可应用于其他生成-验证任务（如程序合成、机器人控制），避免幸存者偏差导致的性能高估。

### 11. 局限性

- 依赖高质量的合成训练数据（约 10 万条轨迹），可能无法完全覆盖真实工业场景的多样性（如复杂装配体、非标准标注）。
- 多轮交互增加计算开销（平均每任务 5-8 轮），实时性有待验证，当前在单 GPU 上推理时间约 30 秒/任务。
- 当前框架主要针对 CadQuery，对其他 CAD 脚本语言（如 OpenSCAD、FreeCAD）的泛化性未知，需额外适配。
- GVPM 中 K 和 η 超参数（K=3, η=0.01）需手动设定，可能影响不同任务的最优性能。

### 12. 我的看法 / 个人笔记

IterCAD 通过闭环迭代和几何感知训练，有效解决了 CAD 自动化中的关键瓶颈，其提出的 CD-TR 指标具有重要方法论价值。GVPM 的信用分配机制是亮点，但合成数据到真实工业场景的迁移仍需验证。未来可探索更高效的交互策略（如提前终止）和跨语言泛化，以及将框架扩展到多部件装配场景。

  为什么重要：展示了语言模型驱动 CAD 自动化的潜力，但存在几何不一致和拓扑缺陷。
  核心贡献：通过生成可执行 Python 代码（CadQuery）构建 3D 模型。
  与当前论文关系：IterCAD 在其基础上引入闭环迭代和几何感知优化，克服了其开环一次性生成的局限，在可执行率上提升 24.2%。

  为什么重要：类似利用程序化 CAD 框架，但缺乏自校正能力。

  与当前论文关系：IterCAD 通过多轮交互和强化学习提升了鲁棒性，在 AUC-TR 上提升 0.29。

### 14. 总结

IterCAD 是一个统一的闭环多模态智能体框架，用于交互式 CAD 生成与编辑。它通过两阶段训练（渐进式 SFT + 几何感知 RL，含 GVPM）和新的评估指标（CD-TR 曲线），在 IterCAD-Bench 上代码可执行率达 92.3%（CAD-Coder 为 68.1%），AUC-TR 达 0.87（CAD-Coder 为 0.52），显著优于现有基线，推动了 CAD 自动化向迭代式工作流程的转变。

## 3. AgentBeats: Agentifying Agent Assessment for Openness, Standardization, and Reproducibility

**Topics:** 智能体评估, 标准化协议, 基准测试智能体化
**Authors:** Xiaoyuan Liu, Jianhong Tu, Yuqi Chen, Siyuan Xie, Sihan Ren, Tianneng Shi, Gal Gantar, Evan Sandoval, Donghyun Lee, Daniel Miao, Peter J. Gilbert, Nick Hynes, Mauro Staver, Warren He, David Marn, Andrew Low, Xi Zhang, Elron Bandel, Michal Shmueli-Scheuer, Siva Reddy, Alexandre Drouin, Alexandre Lacoste, Ramayya Krishnan, Elham Tabassi, Yu Su, Victor Barres, Chenguang Wang, Wenbo Guo, Dawn Song
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.13608v1) | [PDF](http://arxiv.org/pdf/2606.13608v1.pdf)

### 1. 一句话结论

提出Agentified Agent Assessment (AAA)范式，将基准测试本身作为智能体（Judge Agent），通过A2A和MCP等标准化协议与被评估智能体交互，将集成复杂度从N×M降至N+M，实现开放、标准化和可复现的评估。

### 2. 为什么重要

当前智能体评估碎片化严重，基准测试与智能体之间需要大量定制集成，导致测试-生产不匹配和结果不可比。AAA范式通过统一接口解耦评估逻辑和被评估智能体，支持自适应评估和多智能体评估，为构建开放、可复现的评估生态提供了可行路径。

### 3. 研究问题

1. 能否定义一种标准化、与智能体无关的评估接口，使其对智能体内部结构假设最少，同时保持跨基准和智能体的互操作性，并减少测试-生产不匹配？
2. 这种标准能否足够表达力强且轻量，以支持广泛的评估风格和部署约束，实际应用中存在哪些不可避免的限制？

### 4. 背景与问题设定

LLM-based agent systems快速发展，但评估碎片化严重。现有基准测试大多采用固定、以LLM为中心的测试框架，需要大量定制集成，导致测试-生产不匹配，限制了不同智能体设计之间的公平比较。根本问题是缺乏开放、与智能体无关的评估接口。

### 5. 方法概览

- 任务 / 环境: 多种智能体任务环境，包括编码、网页浏览、医疗、多智能体游戏等。
- 变量 / 干预设计: 将基准测试本身智能体化（Agentified Benchmark），作为Judge Agent；Judge Agent内部可包含LLM-as-judge、用户模拟器等子组件，这些子组件也可作为独立A2A服务。被评估的Subject Agent通过标准化协议（A2A用于任务管理，MCP用于工具访问）与Judge Agent交互。Delegator负责选择Judge Agent和Subject Agent并启动评估。A2A和MCP作为示例协议，其他协议也可替代。
- 评价指标: 根据具体基准测试预定义的指标，由Judge Agent在评估后计算并报告。
- 模型比较: 在编码智能体案例研究中，比较了四种代表性编码智能体在三个编码基准上的表现，并与公开记录进行对比。
- 我的理解: AAA的核心思想是将评估本身视为一个智能体任务，通过标准化协议解耦评估逻辑和被评估智能体。Judge Agent内部可包含LLM-as-judge等子组件，这些子组件也可作为独立A2A服务。A2A和MCP是示例协议，其他协议也可替代。任何符合协议的智能体都可以被任何符合协议的基准测试评估，无需定制集成。

### 6. 关键发现

- AAA范式在社区规模竞赛中验证了覆盖性和实用性: 为期五个月的开放竞赛吸引了298个Judge Agent（覆盖12个类别）和467个Subject Agent，来自独立开发者，使用多种编程语言，许多评估逻辑通过自然语言提示声明，表明采用门槛低。
- AAA评估在编码智能体案例中保持了保真度: 与公开记录相比，AAA标准化评估结果匹配良好，同时揭示了之前缺失的面对面比较结果，并发现了模型与其原生测试框架之间的共适应效应。
- AAA将集成复杂度从N×M降至N+M: 传统评估需要为每个智能体-基准对进行定制集成，而AAA只需每个智能体和每个基准分别实现一次协议适配即可互操作。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: AAA范式使得任何符合A2A和MCP协议的智能体都可以被标准化评估，无需为每个基准定制集成，从而加速智能体系统的开发和迭代。
- 对 Skill: 标准化评估接口有助于公平比较不同智能体的技能水平，并可能通过自适应评估（如根据早期表现动态调整任务难度）更高效地评估技能。
- 对 Memory: AAA本身不直接涉及记忆机制，但标准化协议支持多轮交互，未来可扩展用于评估智能体的长期记忆和上下文利用能力。
- 对评测: AAA将评估从固定数据集+评分函数转变为交互式评估过程，支持自适应评估、多智能体评估，并提高了评估的可复现性和互操作性。

### 11. 局限性

- AAA假设智能体必须兼容A2A和MCP协议（作为示例协议），对于不支持这些协议的现有智能体需要额外适配。
- 语义内化（用自然语言描述评估逻辑）可能引入LLM-as-judge的偏差，影响评估保真度。
- 目前仅在编码智能体案例中进行了保真度验证，其他类型基准的保真度有待进一步研究。

### 12. 我的看法 / 个人笔记

AAA范式通过将评估本身智能体化并复用生产级协议，优雅地解决了智能体评估碎片化问题。社区竞赛和编码案例提供了有力的实证支持。未来工作应关注协议兼容性扩展、语义内化偏差的量化，以及自适应评估策略的优化。

### 14. 总结

本文提出Agentified Agent Assessment (AAA)范式，将基准测试本身作为Judge Agent，通过A2A和MCP等标准化协议与被评估的Subject Agent交互，实现评估的开放、标准化和可复现。AgentBeats系统实例化了AAA，并定义了五种操作模式以适应实际约束。通过五个月开放竞赛和编码智能体案例研究，验证了AAA在覆盖性、实用性和保真度方面的有效性，为构建统一的智能体评估生态提供了可行方案。

## 4. Multi-Agent Reinforcement Learning from Delayed Marketplace Feedback for Objective-Weight Adaptation in Three-Sided Dispatch

**Topics:** Multi-Agent Reinforcement Learning, Offline Reinforcement Learning, Dispatch Optimization, Three-Sided Marketplace
**Authors:** Haochen Wu, Yi Hou, Shiguang Xie
**Venue / Year:** ICML Workshop on RLxF, 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.13604v1) | [PDF](http://arxiv.org/pdf/2606.13604v1.pdf)

### 1. 一句话结论

DoorDash部署了一个离线多智能体强化学习系统，通过集中训练-分散执行框架调整调度优化器的目标权重，在不降低客户体验的前提下提高了批处理效率并减少了配送员时间成本。

### 2. 为什么重要

该工作展示了如何利用延迟、噪声和耦合的市场反馈信号，在真实生产环境中安全地调整大规模物流系统的决策策略，为强化学习在复杂经济系统中的应用提供了可落地的范例，并强调了多智能体设置和离线训练技术的重要性。

### 3. 研究问题

1. 如何利用延迟、噪声和耦合的市场反馈信号来学习调度策略？
2. 在不替换现有组合优化器的情况下，如何通过调整目标权重来改善配送效率？
3. 如何设计多智能体框架，使每个商店作为独立智能体，通过集中离线训练和分散执行实现自适应决策？

### 4. 背景与问题设定

食品配送调度是一个三方市场（顾客、商家、配送员）中的序贯决策问题，需要平衡配送速度、配送员利用率和商家拥堵等多个目标。传统方法使用静态启发式权重，难以适应局部和时变条件。本文提出一个离线多智能体强化学习系统，通过学习一个策略来调整调度优化器的目标权重，从而在保持生产可行性和操作安全的前提下实现自适应。每个商店被视为一个独立智能体，使用集中式离线训练和分散式执行。

### 5. 方法概览

- 任务 / 环境: DoorDash的食品配送调度系统，包含一个组合分配优化器（内层）和一个目标权重调整代理（外层）。每个商店作为一个智能体，在每次调度周期观察本地状态（待配送订单数、供应压力、配送员等待时间）并选择一个离散乘数（0.8, 0.9, 1.0, 1.1, 1.2）来调整优化器的ASAP权重。
- 变量 / 干预设计: 策略选择离散乘数作用于基线ASAP权重，从而影响优化器在配送速度和批处理效率之间的权衡。训练使用离线数据，包括两个迭代的数据收集：纯探索和混合策略（50%利用+50%探索）。使用Double DQN和保守Q学习（CQL）正则化来减少分布外价值高估。
- 评价指标: 效率指标：配送员活跃时间（CAT）、配送员等待时间（CWT）、批处理百分比；质量指标：ASAP（客户端配送时长）、20分钟晚点率。
- 模型比较: 与DoorDash生产基线（静态目标权重）进行全局切换实验对比，使用CUPED方差缩减和聚类p值进行统计检验。
- 我的理解: 该方法通过一个低维控制接口（目标权重乘数）来引导现有优化器，而不是直接替换它，从而保证了系统的稳定性和可部署性。多智能体设置允许每个商店根据本地状态独立决策，而集中训练则利用全局数据学习共享价值函数。Double DQN和CQL正则化是离线训练的关键，前者减少最大化偏差，后者抑制对弱支持动作的高估。

### 6. 关键发现

- 离线训练策略在线上实验中提升了效率: 在所有时段，OWA-RL策略显著降低了CAT（-1.261秒，p=0.019）和CWT（-0.856秒，p=0.004），并将批处理率提高了0.495个百分点（p<0.001），同时ASAP和20分钟晚点率无显著变化。
- 晚餐时段效果更显著: 在晚餐时段，CAT降低1.289秒（p=0.042），CWT降低1.030秒（p=0.041），批处理率提高0.600个百分点（p=0.010），且20分钟晚点率显著改善（-0.037个百分点，p=0.040）。
- 策略表现出状态依赖的自适应行为: 在旧金山湾区周五晚餐高峰期间，策略根据待配送订单数、供应压力和配送员等待时间动态调整ASAP权重乘数，而非固定全局调优。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: 展示了多智能体强化学习在真实物流系统中的应用，通过集中式离线训练和分散式执行，实现了可扩展的自适应决策。每个商店作为独立智能体，利用本地状态进行决策，同时共享全局价值函数。

- 无模型达到50%端点通过率: 最佳模型GPT-5.5/Pi仅通过45.0%的尝试（143/318，95% CI: 36.3-53.7），且领先模型间置信区间重叠，表明当前智能体在表观基因组学分析中整体不可靠。
- 组件级字段正确率与端点通过率存在巨大差距: 在26574个评分字段中，68.2%通过（18124/26574），但端点通过率仅31.0%（1578/5088），说明智能体常找到正确文件并计算有用中间值，但最终选择错误答案。该对比来自Figure 4及相关分析。
- 失败源于错误科学选择而非工具使用: 手动审查25个评估发现，失败常因使用错误统计量、错误阈值、错误单位、错误处理特征或数据层、错误处理读段或峰值表示，以及依赖文献先验而非文件证据。具体案例包括：CUT&RUN中使用端到端比对而非局部比对导致spike-in标准化错误；WGBS中将CpG二核苷酸的两条Bismark行视为独立位点改变覆盖度和效应量；ATAC-seq中直接使用双端BAM而非单碱基Tn5插入位点表示导致峰值偏移。
- 不同实验类型表现差异: CUT&Tag/CUT&RUN通过率最高（34.0%，768/2256），甲基化-seq次之（33.3%，400/1200），ChIP-seq为30.6%（147/480），ATAC-seq最低（22.8%，263/1152）。但差异与任务组成相关，非单纯实验难度。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: 智能体需增强实验特异性推理能力，而非仅依赖通用工作流或文献先验；应集成数据驱动决策机制，避免在已有正确中间结果时替换为错误答案。
- 对 Skill: 需要培养智能体对实验细节（如比对模式、计数策略、数据表示）的敏感性，以及从文件证据中推导结论的能力。
- 对 Memory: 智能体应避免过度依赖训练数据中的常见模式，而应基于当前数据上下文动态调整。
- 对评测: 验证性基准应包含组件级评分以区分工具执行与科学判断，并覆盖多样化实验类型以暴露领域特定失败。

### 11. 局限性

- 评估任务在实验类型和任务类别上不平衡（CUT&Tag/CUT&RUN占47项，ATAC-seq仅24项），可能影响结论普适性。
- 部分失败模式在相关评估中重复出现，导致聚合分数部分反映重复测试。
- 确定性评分器无法覆盖所有科学上合理的分析路径，基准仅覆盖可验证的决策子集。

### 12. 我的看法 / 个人笔记

该基准设计精巧，通过短时程决策点有效分离了工具操作与科学判断，组件级评分与端点通过率的对比尤其揭示了当前AI智能体的根本局限。未来改进应聚焦于增强智能体对实验上下文的理解和证据驱动的推理能力，而非单纯扩大模型规模。

### 14. 总结

EpiBench是一个针对短时程表观基因组学分析的可验证基准，包含106个评估任务和5088条轨迹。结果显示，最佳模型GPT-5.5/Pi仅通过45.0%的尝试，组件级字段正确率（68.2%）与端点通过率（31.0%）的巨大差距表明失败主因是智能体选择文献先验或通用工作流而非文件证据。该基准为评估和改进生物信息学AI智能体提供了重要工具。
