## 1. MAS-PromptBench: When Does Prompt Optimization Improve Multi-Agent LLM Systems?

**Topics:** Multi-Agent Systems, Prompt Optimization, LLM Agents
**Authors:** Juyang Bai, Laixi Shi
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.23664v1) | [PDF](http://arxiv.org/pdf/2606.23664v1.pdf) | [Code](https://github.com/juyangbai/MAS-PromptBench)

### 1. 一句话结论

Prompt optimization can significantly improve multi-agent LLM systems, but gains vary widely by task, topology, and team size, with risks of performance degradation.

### 2. 为什么重要

This work provides the first systematic benchmark to quantify when and how much prompt optimization helps in multi-agent systems, guiding practitioners and algorithm designers.

### 3. 研究问题

1. How much can prompt optimization improve multi-agent LLM system performance?
2. How do task domain, workflow topology, communication protocol, and team size affect optimization gains?

### 4. 背景与问题设定

Multi-agent systems (MAS) with LLM-based agents rely on system prompts to define roles and behaviors. While prompt optimization is effective for single LLMs, extending it to MAS faces combinatorial explosion due to inter-agent dependencies. Current MAS frameworks use manual prompts, and the potential gains from automatic optimization are unknown.

### 5. 方法概览

- 任务 / 环境: Nine tasks across reasoning (GPQA-Diamond, HotpotQA, MATH), coding (LiveCodeBench, APPS, SWE-Bench), and tool-calling (BFCL, ToolHop, API-Bank).
- 变量 / 干预设计: Systematic variation of four MAS configuration factors: task, workflow topology (Single, Independent, Sequential, Centralized, Decentralized), communication protocol (Freeform, Semi-structured, Structured), and team size (2,4,8,10).
- 评价指标: Prompt-optimization gain Δ = performance(optimized prompts) - performance(default prompts) in percentage points, using official evaluation metrics per task.
- 模型比较: Two multi-agent extensions of single-agent optimizers: MAS-GEPA (evolutionary) and MAS-MIPRO (programmatic).
- 我的理解: The benchmark isolates the effect of prompt optimization by fixing all other MAS components, enabling controlled comparisons across configurations.

### 6. 关键发现

- Prompt optimization yields large gains but also risks: Improvements up to +24.0 percentage points (Sequential on BFCL), but drops as severe as -16.0 points occur, indicating need for tailored algorithms.
- Gains depend on task and topology: Average gains positive on 7/9 tasks; largest average gain +10.0 points on APPS. Sequential topology shows highest gains, while Independent can degrade performance.
- Communication protocol and team size matter: Structured protocols (e.g., structured communication) facilitate optimization; larger teams (n=10) reduce gains, confirming scaling challenges.

### 7. 最重要的图

- 图来源: Figure 1 in paper
- 为什么重要: Illustrates the stark contrast between consistent single-agent gains and highly variable multi-agent gains across tasks and topologies, motivating the study.

### 8. 如何读这张图

Bar chart showing percentage point changes from prompt optimization. Single-agent bars are all positive; multi-agent bars vary from +14 to -16, highlighting instability.

### 9. 次重要图

- 为什么也重要: Figure 2 overviews the benchmark's modular design, clarifying the four configuration axes (task, topology, protocol, team size) and their combinations.

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: Prompt optimization should be topology-aware; sequential workflows benefit most, while independent topologies may require different strategies.
- 对 Skill: Optimizing agent-local behaviors (e.g., tool-calling) yields larger gains than holistic reasoning tasks.
- 对 Memory: Not directly addressed, but structured communication protocols may aid memory sharing across agents.
- 对评测: Benchmark provides controlled protocol to isolate prompt effects; future evaluations should consider configuration sensitivity.

### 11. 局限性

- Only two prompt optimizers evaluated; results may not generalize to all methods.
- Team size limited to 10; larger teams may show different dynamics.
- Communication protocols are predefined; adaptive protocols not studied.

### 12. 我的看法 / 个人笔记

This benchmark is a crucial step toward principled MAS design, revealing that prompt optimization is not a silver bullet but a configuration-dependent lever. The observed performance drops highlight the need for robust, scalable algorithms that account for inter-agent dependencies.

### 13. 这篇工作引用的近期高影响力相关文章

- AutoGen: Enabling Next-Gen LLM Applications via Multi-Agent Conversation
  为什么重要：Foundational MAS framework using manual prompts; current work automates prompt optimization.
  核心贡献：Open-source framework for multi-agent conversations.
  与当前论文关系：Current work evaluates prompt optimization on AutoGen-based topologies.
- DSPy: Compiling Declarative Language Model Calls into Self-Improving Pipelines
  为什么重要：Single-agent prompt optimization framework; extended to multi-agent in this work.
  核心贡献：Optimizes prompts within multi-stage LLM programs.
  与当前论文关系：MAS-MIPRO optimizer is adapted from DSPy's MIPROv2.
- PromptAgent: Strategic Planning with Language Models Enables Expert-level Prompt Optimization
  为什么重要：State-of-the-art single-agent optimizer; basis for MAS-GEPA.
  核心贡献：Planning-based prompt optimization via tree search.
  与当前论文关系：MAS-GEPA extends this evolutionary approach to multi-agent settings.

### 14. 总结

MAS-PromptBench systematically evaluates prompt optimization in multi-agent LLM systems across tasks, topologies, protocols, and team sizes. Results show potential gains up to +24 points but also risks of degradation, emphasizing the need for topology-aware and scalable optimization algorithms.

## 2. AIR: Adaptive Interleaved Reasoning with Code in MLLMs

**Topics:** Multimodal Large Language Models, Reinforcement Learning, Tool Use, Numerical Computation
**Authors:** Cong Han, Xiaohan Lan, Haibo Qiu, Yujie Zhong
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.23678v1) | [PDF](http://arxiv.org/pdf/2606.23678v1.pdf) | [Code](https://github.com/CongHan0808/AIR.git)

### 1. 一句话结论

AIR通过强化学习训练，使多模态大模型在复杂数值计算任务中自适应地交织文本推理与代码执行，性能平均提升6.1个百分点，工具调用成功率超95%。

### 2. 为什么重要

现有MLLMs在工具使用上多局限于视觉操作（如旋转、缩放），缺乏处理复杂数值计算的能力。AIR首次实现了模型在推理过程中自适应调用代码工具，显著提升了数学推理的准确性和稳定性，并解决了Agentic RL中因工具调用比例增加导致的训练不稳定甚至模型崩溃问题。

### 3. 研究问题

1. 如何使MLLMs在复杂数值计算任务中自适应地决定是否以及何时调用代码工具？
2. 如何通过强化学习训练实现稳定且高效的交互式推理，避免因工具调用比例增加导致的训练崩溃？

### 4. 背景与问题设定

随着OpenAI o3等模型的发布，交织推理与代码成为增强MLLMs的前沿方向。然而，现有工作主要聚焦于视觉感知任务中的工具使用（如旋转、缩放），依赖预定义启发式规则，无法处理需要精确数值计算的问题。本文旨在通过强化学习训练，使MLLMs在数学推理中自主决定是否调用Python代码，以提升计算精度和推理能力，同时解决Agentic RL中因工具调用比例增加导致的训练不稳定问题。

### 5. 方法概览

- 任务 / 环境: 多模态数学推理任务，输入为图像和问题，模型需生成最终答案。支持在推理过程中调用Python代码，代码在沙箱环境中执行并返回结果。
- 变量 / 干预设计: 两阶段训练：冷启动SFT（使用两阶段数据构建管道：先由MLLM生成纯文本CoT，再由LLM重写为交织推理轨迹，经代码执行验证和结果准确性过滤）和强化学习（使用GRPO算法，并引入组约束奖励函数，限制组内交织推理序列的比例）。数据过滤策略包括自采样策略（基于模型自身多次roll-out的成功率）和先验过滤策略（利用强LLM判断样本是否需要工具调用）。
- 评价指标: 在评估基准上的平均性能提升（6.1个百分点）、交织推理样本准确率提升（9.9个百分点）、工具调用成功率（>95%）。
- 模型比较: 与基线模型（未使用交织推理或未经过RL训练）对比，AIR在数值计算任务上显著提升。具体比较了使用不同数据过滤策略和奖励函数的变体，并报告了组约束奖励函数对训练稳定性的影响。
- 我的理解: 该方法的核心在于通过两阶段数据构建降低数据生成难度，并通过组约束奖励函数在RL训练中平衡工具调用与文本推理，避免模型过度依赖工具或训练不稳定。实验表明，当工具调用比例增加时，无组约束的训练会出现不稳定甚至模型崩溃，而组约束有效缓解了这一问题。

### 6. 关键发现

- 自适应工具调用提升推理性能: 经过RL训练后，模型在评估基准上平均提升6.1个百分点，交织推理样本准确率提升9.9个百分点，工具调用成功率超过95%。
- 组约束奖励函数确保训练稳定性: 在Agentic RL中，随着工具调用比例增加，训练可能变得不稳定甚至导致模型崩溃。组约束奖励函数通过限制组内交织推理序列的比例，有效解决了这一问题，实现了长期稳定训练。实验观察到，无组约束时模型在训练后期出现性能骤降，而组约束保持了稳定提升。
- 冷启动数据构建方法有效: 两阶段数据构建管道（先文本CoT再交织推理重写）显著降低了生成高质量交织推理数据的难度，验证和过滤步骤确保了数据的正确性和可用性。

- 图来源: Figure 1 (左图：AIR概览；右图：组约束奖励设计)
- 为什么重要: 该图直观展示了AIR的整体架构：模型接收图像和问题后，生成文本推理并可选地调用代码工具，代码在沙箱中执行后反馈结果，循环直至输出最终答案。右图则说明了组约束奖励如何计算组相对优势，这是实现自适应工具调用和训练稳定性的关键。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: AIR展示了如何通过RL训练使智能体在推理过程中自适应调用外部工具，为构建更灵活的Agent系统提供了方法，特别是在需要精确计算的场景中。
- 对 Skill: 模型学会了在文本推理和代码执行之间自主切换，这种技能组合对于处理复杂多步任务至关重要。
- 对 Memory: 交互式推理过程中，代码执行结果被反馈并融入后续推理，相当于一种外部记忆机制，减轻了模型对内部记忆的依赖。
- 对评测: 提出了新的评估维度：工具调用成功率、交织推理准确率，以及训练稳定性，为评估Agentic RL模型提供了更全面的指标。

### 11. 局限性

- 实验主要基于数学推理任务，未在更广泛的多模态任务（如视觉问答、图表理解）中验证。
- 冷启动数据构建依赖强MLLM（如Gemini）和LLM，可能引入数据偏差和成本问题。
- 组约束奖励函数中的阈值δ需要手动调整，可能影响泛化性。

### 12. 我的看法 / 个人笔记

AIR在MLLMs的数值计算推理方面迈出了重要一步，其自适应工具调用和训练稳定性解决方案具有实用价值。未来工作可探索将类似方法扩展到更多工具类型（如搜索、数据库查询）和更复杂的多步任务。

### 14. 总结

AIR提出了一种通过强化学习训练使MLLMs在复杂数值计算任务中自适应交织文本推理与代码执行的方法。通过两阶段冷启动数据构建、数据过滤策略和组约束奖励函数，模型在评估基准上平均提升6.1个百分点，工具调用成功率超过95%，并解决了Agentic RL中因工具调用比例增加导致的训练不稳定问题。

## 3. TriggerBench: Investigating Prospective Memory for Large Language Models

**Topics:** Prospective Memory, Large Language Models, Benchmark, Cognitive Science
**Authors:** Tianhua Zhang, Xinjiang Wang, Qianxi Zhang, Qi Chen, Kun Li, Yaoqi Chen, Dingdong Wang, Helen Meng, Yan Lu
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.23459v1) | [PDF](http://arxiv.org/pdf/2606.23459v1.pdf) | [Code](https://github.com/KristenZHANG/TriggerBench-Official)

### 1. 一句话结论

TriggerBench reveals that LLMs' prospective memory (spontaneous recall of latent constraints) is significantly weaker than retrospective memory, degrades sharply with context length, and can serve as a behavioral probe of spare reasoning capacity.

### 2. 为什么重要

As LLMs are deployed in long interactions, the ability to proactively recall and act on latent constraints (prospective memory) is critical for safety and reliability, yet existing evaluations focus on reactive retrieval. This work provides the first comprehensive benchmark and reveals fundamental limitations in current models.

### 3. 研究问题

1. How well can LLMs spontaneously recall and act on latent constraints (prospective memory) without explicit prompts?
2. How does prospective memory compare to retrospective memory under identical contexts?
3. Can prospective memory serve as a behavioral probe of spare reasoning capacity?

### 4. 背景与问题设定

LLMs are increasingly used in long interactions, but evaluations focus on retrospective memory (explicit query-based retrieval). Prospective memory—the ability to spontaneously recall and act on latent constraints—remains unevaluated. Real-world failures (e.g., forgetting a cello in a taxi) illustrate the need for this capability. TriggerBench addresses this gap with a benchmark spanning five dimensions across daily and professional workflows.

### 5. 方法概览

- 任务 / 环境: Multi-turn dialogues simulating daily assistant and professional workflow scenarios, with latent constraints introduced early and triggers appearing later after orthogonal turns.
- 变量 / 干预设计: Three contrastive variants: Positive-Clean (standard), Positive-Overloaded (trigger with distracting user request), Negative-Clean (risk resolved, testing false alarms). Also includes matched retrospective memory probes.
- 评价指标: Slot Match (entity recall) and PM Accuracy (LLM-based evaluation of proactive intent fulfillment).
- 模型比较: Evaluates open-source (Qwen3, Gemma3) and closed-source (GPT-4o, GPT-4.1, GPT-5.2) LLMs, plus RAG and memory systems (A-MEM, Mem0, Letta-Sim).
- 我的理解: The benchmark cleverly decouples constraints from triggers using orthogonal turns to prevent lexical shortcuts, and includes negative controls to detect 'always-remind' heuristics. The parallel RM probes isolate retention failures from reasoning failures.

### 6. 关键发现

- Precision-Recall Trade-off and Attentional Fragility: Enhanced reasoning improves proactive recall but may lead to an 'always-remind' heuristic (high false alarms). PM accuracy degrades under implicit constraints or overloaded triggers, indicating attentional fragility.
- PM is Harder than RM: On identical contexts, RM near-saturates up to 100K tokens, while PM decays sharply with context length, revealing a 'cognitive cliff'.
- PM as a Probe of Spare Reasoning Capacity: Pairing PM with math problems shows successful reasoning trajectories yield higher PM accuracy than failed ones at the same context length, indicating PM tracks spare reasoning budget.

### 7. 最重要的图

- 图来源: Figure 1: Overview of TriggerBench
- 为什么重要: Illustrates the benchmark structure: constraint-trigger decoupling, orthogonal turns, and contrastive variants, which are key to evaluating prospective memory.

### 8. 如何读这张图

The figure shows a dialogue timeline: a constraint (cello in trunk) is established early, followed by orthogonal turns, then a trigger (user exiting taxi). The model must proactively remind. Variants include negative resolution (driver reminds) and overloaded distraction (buying flowers).

### 9. 次重要图

- 为什么也重要: Table 3 (Main Results) shows the performance gap between PM and RM, the precision-recall trade-off, and the impact of context scaling, providing quantitative evidence for key findings.

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: Proactive agents must be designed with robust prospective memory, not just retrieval; attention hijacking and false alarms are critical failure modes.
- 对 Skill: Reasoning enhancement improves PM but may introduce bias; training should balance recall and precision.
- 对 Memory: Active working memory (e.g., Letta-Sim's core memory) is crucial for PM; retrieval-based systems fail due to semantic gap.
- 对评测: PM benchmarks should include negative controls and overloaded variants to avoid misleading metrics.

### 11. 局限性

- Benchmark focuses on text-based dialogues; real-world multimodal scenarios (e.g., GUI agents) are not covered.
- Evaluation relies on LLM-as-judge, which may have biases; human validation is limited.
- Only 488 blueprints; scaling to more diverse domains may be needed.

### 12. 我的看法 / 个人笔记

TriggerBench is a timely and well-designed benchmark that highlights a critical blind spot in LLM evaluation. The finding that PM tracks spare reasoning capacity is particularly insightful, suggesting PM could be a diagnostic tool for cognitive load. The precision-recall trade-off warns against over-optimizing for proactivity without considering false alarms.

### 13. 这篇工作引用的近期高影响力相关文章

- LoCoMo: Long Context Memory Evaluation
  为什么重要：Evaluates retrospective memory in multi-session dialogues.
  核心贡献：Focuses on factual retrieval via explicit queries.
  与当前论文关系：TriggerBench extends to prospective memory and includes matched RM probes for comparison.
- Mittal (2026): Prospective Memory Failures under Cognitive Load
  为什么重要：Explores PM but uses explicit formatting rules.
  核心贡献：Studies PM under cognitive load with delegated instructions.
  与当前论文关系：TriggerBench assesses context-driven situational awareness without explicit delegation.
- LoCoMo-Plus: Latent Constraint Retention
  为什么重要：Extends to latent constraints under semantic disconnects.
  核心贡献：Evaluates retention of implicit constraints.
  与当前论文关系：TriggerBench further requires proactive intervention upon triggers.

### 14. 总结

TriggerBench introduces a comprehensive benchmark for evaluating prospective memory in LLMs, revealing that models struggle with spontaneous recall of latent constraints, especially under long contexts and distractions. Enhanced reasoning improves recall but risks false alarms. PM is harder than RM and can serve as a behavioral probe of spare reasoning capacity. The work underscores the need for robust prospective memory in deployed LLMs.

## 4. EnterpriseClawBench: Benchmarking Agents from Real Workplace Sessions

**Topics:** GUI Agent, Enterprise Agent Benchmark, Multidimensional Evaluation
**Authors:** Jincheng Zhong, Weizhi Wang, Che Jiang, Kai Tian, Zhenzhao Yuan, Junlin Yang, Dianqiao Lei, Kaiyan Zhang
**Venue / Year:** 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.23654v1) | [PDF](http://arxiv.org/pdf/2606.23654v1.pdf) | [Code](https://github.com/FrontisAI/EnterpriseClawBench)

### 1. 一句话结论

EnterpriseClawBench从真实企业会话构建852个可复现任务，Codex/GPT-5.5组合在120任务Lite子集上仅得0.663分，揭示企业智能体评估需多维度报告而非单一分数。

### 2. 为什么重要

该基准填补了企业智能体评估中真实性与可扩展性之间的空白，强调评估必须考虑框架-模型组合、工件交付、视觉质量、成本、运行时和技能迁移行为，而非仅依赖模型分数。

### 3. 研究问题

1. 如何从真实企业会话中自动构建可复现的基准任务？
2. 企业智能体评估应包含哪些关键维度以全面反映性能？
3. 技能迁移效果如何依赖于创建者模型和消费者框架？

### 4. 背景与问题设定

企业智能体在可执行工作空间中操作，需读取异构文件、调用工具并交付业务工件。现有基准存在三个缺口：任务构建的真实性与可扩展性矛盾、评估维度单一、缺乏任务类级别的技能评估。

### 5. 方法概览

- 任务 / 环境: 基于超过100名员工的企业AI系统内部会话（2026年3月至5月），在持久化Linux沙箱中运行，输入挂载至input目录，输出写入output目录。
- 变量 / 干预设计: 从5,291个原始任务实例经长度过滤、夹具查找、脱敏恢复、网络依赖检查等机械门控，最终得到852个可复现任务，并附加7个角色类、45个技能子类、硬规则和语义评分标准。
- 评价指标: 双层评分：硬规则检查（文件类型、数量、非空、可打开性等）和语义评分（五个维度：基础准确性、任务相关性、实质深度、实用性和沟通质量），文本输出用文本裁判，HTML/幻灯片等用视觉裁判。
- 模型比较: 评估32种框架-模型组合，包括Claude Code、Codex、DeepAgents、Hermes和OpenClaw，搭配GPT-5.5、Sonnet 4.6、Opus 4.6等模型。
- 我的理解: 该基准通过自动化管道将真实会话转化为标准化任务，并引入多维评估框架，强调框架与模型的耦合效应，以及技能迁移的评估。

### 6. 关键发现

- 最佳性能仅0.663分: 在120任务的Lite子集上，Codex/GPT-5.5组合取得最高分0.663，表明企业工件任务远未饱和。
- 框架-模型耦合显著影响性能: Claude系列模型在Hermes框架下性能大幅下降（如Sonnet 4.6从0.62-0.64降至0.458），源于Hermes的审批检查频繁阻断脚本执行和多步修复，导致工件未写入/workspace/outputs目录。
- 成本与分数呈对数关系: 从低成本到中等成本系统分数提升显著，但中等成本以上边际收益递减；Hermes/Claude组合因框架不匹配成为异常值（高成本低分数）。
- 角色类与工件类型影响模型排名: GPT-5.5在多数角色类中表现最佳，但营销和财务/运营类任务较难；文件格式改变排名，视觉裁判对幻灯片和电子表格存在分数膨胀。
- 技能迁移依赖创建者模型: 针对前端页面生成子类，GPT-5.5作为技能创建者产生正迁移（平均+0.068），而Haiku 4.5导致负迁移（平均-0.094），技能注入效果高方差。

## 5. Causal Discovery in the Era of Agents

**Topics:** GUI Agent, Causal Discovery, Large Language Models
**Authors:** Yujia Zheng, Vishal Verma, Mantej Gill, Haoyue Dai, Peter Spirtes, Kun Zhang
**Venue / Year:** arXiv 2026
**Links:** [Abstract](http://arxiv.org/abs/2606.23608v1) | [PDF](http://arxiv.org/pdf/2606.23608v1.pdf) | [Code](https://causallearn.com)

### 1. 一句话结论

大型语言模型应辅助因果发现流程，而非直接提供因果证据；因果结论必须基于数据、明确假设和形式化算法。

### 2. 为什么重要

该论文提出了一个关键原则：在因果发现中，智能体（如LLM）只能辅助工作流，不能提供因果证据。这避免了将文本关联、提示伪影或幻觉机制误认为因果结论，确保了因果发现的可靠性和可追溯性。

### 3. 研究问题

1. 在因果发现中，大型语言模型应扮演什么角色？
2. 如何设计一个既利用LLM能力又保持因果结论可靠性的系统？

### 4. 背景与问题设定

因果发现旨在从观测数据中学习因果关系，但现有方法依赖强假设，对非专业用户困难。近期工作将LLM引入因果发现，让模型推断方向、提出图结构或注入先验，但这模糊了因果证据的来源——可能来自数据、文本关联、提示伪影或幻觉。论文主张智能体应辅助工作流，而非替代因果发现。

### 5. 方法概览

- 任务 / 环境: 因果发现任务，用户上传数据（如大五人格问卷数据），通过平台进行数据理解、预处理、方法推荐、专家知识融入、形式化发现和解释。
- 变量 / 干预设计: 设计causal-learn+平台，智能体围绕形式化因果发现核心（如PC、FCI、GES、LiNGAM等算法）提供辅助，但不参与算法内部计算。用户必须明确批准预处理、假设和方法选择。
- 评价指标: 通过案例研究（大五人格数据）展示平台可用性，强调因果图的可解释性和与领域知识的对齐。
- 模型比较: 对比现有LLM直接参与因果发现的方法（如LLM推断方向、注入先验），指出其模糊证据来源的问题。
- 我的理解: 该方法通过将智能体限制在辅助角色，确保了因果结论的可靠性。智能体提供上下文、解释和工具协调，但因果图完全由形式化算法基于数据和用户批准的假设生成。

### 6. 关键发现

- 智能体不能提供因果证据: LLM输出可能来自文本关联、提示伪影或幻觉，若直接用于图构建，因果结论的证明状态变得模糊。因果证据必须来自数据、明确假设、形式化算法、诊断和用户/领域专家决策。
- 智能体的正确角色是辅助工作流: 智能体应帮助数据理解、预处理、方法选择、专家知识融入、工具协调和结果解释，但不应提供边、方向、先验、约束或因果结论。
- causal-learn+平台实现了该原则: 平台将分析分为数据检查、假设引出、方法推荐、工具协调、算法执行、图检查和结果解释等可追溯步骤，智能体支持这些步骤，但因果发现核心由causal-learn算法执行。

### 10. 对 Agent / Skill / Memory 的启发

- 对 Agent 系统: 智能体系统应明确区分辅助和推理角色，避免将LLM输出误认为因果证据。设计应确保用户对关键决策有控制权。
- 对 Skill: 智能体需要具备数据理解、方法推荐、工具协调和结果解释等技能，但不应具备因果推理能力。
- 对 Memory: 智能体应记录用户决策和假设，确保可追溯性，但不应将记忆中的知识直接用作因果约束。
- 对评测: 评估智能体系统时，应关注其辅助效果（如减少用户错误、提高效率），而非其因果推理准确性。

### 11. 局限性

- 论文未提供大规模用户研究来验证causal-learn+的可用性和有效性。
- 案例研究仅基于大五人格数据，未涵盖其他领域（如基因组学、神经科学）。
- 智能体辅助的专家知识融入仍依赖LLM的检索能力，可能引入不准确信息。

### 12. 我的看法 / 个人笔记

该论文提出了一个重要的设计原则：在因果发现中，智能体应辅助而非替代。这有助于防止LLM的不可靠性污染因果结论。causal-learn+平台是这一原则的务实实现，但未来需要更多实证研究来验证其在实际工作流中的效果。
